# Dev Studio Website - Performance Audit Report

## ЁЯФ┤ CRITICAL (causes jank / blocking render)

- **`app/components/WorkSection.tsx` (Line 96-143):**
  - **Issue:** The GSAP ScrollTrigger configuration for the horizontal scroll is missing `.revert()` cleanup in the `useLayoutEffect` return block for the matched media sections. The current cleanup only removes event listeners and kills specific animations, but `ctx.revert()` is the proper way to clean up GSAP contexts in React 18+. Although `ctx.revert()` is called at the end of `useLayoutEffect`, it might not cover dynamically created animations inside `mm.add` perfectly depending on how GSAP manages it internally; prefer using `revert()` on `mm` instead of or in addition to `ctx` more explicitly.
  - **Fix:** Ensure `mm.revert()` and `ctx.revert()` are comprehensively covering all ScrollTrigger instances.

- **`app/components/WorkSection.tsx` (Line 298-300):**
  - **Issue:** The horizontal scrolling `.work-track` container uses `will-change: transform`. While this is good, having it statically applied to the CSS might cause the browser to persistently retain a large compositor layer in memory.
  - **Fix:** Apply `will-change: transform` dynamically only when the horizontal scroll is active (e.g., via GSAP) or keep it if memory profiling shows no issues, but monitor it closely.

- **`app/components/HeroSection.tsx` & `components/ui/SplineScene.tsx`:**
  - **Issue:** `@splinetool/react-spline` is heavily impacting the initial load. It is not dynamically imported, making it block the main thread during hydration.
  - **Fix:** Wrap `SplineScene` in a `next/dynamic` import with SSR disabled (`ssr: false`) and a lightweight placeholder.

- **`components/globe.tsx` (Line 3):**
  - **Issue:** `cobe` globe component is loaded synchronously. Given it renders on a canvas and has an animation loop, this can block the main thread early on.
  - **Fix:** Import `Earth` component using `next/dynamic` where it is used.

## ЁЯЯб MODERATE (causes slowness, not jank)

- **Missing `next/dynamic` for heavy components:**
  - `components/ui/DashboardScreen.tsx`
  - `components/globe.tsx`
  - **Fix:** Use `import { dynamic } from 'next/dynamic'` for these components, especially those that appear "below the fold" or perform heavy canvas/SVG animations.

- **`components/ui/DashboardScreen.tsx` (Line 15-37):**
  - **Issue:** Uses `gsap.to({})` to animate React state values (`setProjects`, `setClients`, `setDays`). This causes a React re-render for every frame of the GSAP animation, which is highly inefficient.
  - **Fix:** Use standard React tools for animating numbers (e.g., Framer Motion's `useSpring` and `useTransform` with `<motion.span>`) or use GSAP to directly manipulate the DOM nodes via refs instead of setting state.

- **`app/components/ProjectCard.tsx`:**
  - **Issue:** `ProjectCard` component renders complex Framer Motion transitions and image loading for every card concurrently in `WorkSection`.
  - **Fix:** Wrap the `ProjectCard` content or mapping in a `React.memo` if props don't change, or use `lazyMotion` from framer-motion to reduce the main bundle size.

- **Missing `next/image`:**
  - **`app/components/WhyChooseUs.tsx` (Line 333, 375):** Marquee SVG logos are using basic `<img src={...}>`.
  - **Fix:** Migrate to `<Image>` from `next/image`, or continue using `<img>` but ensure explicit `width` and `height` attributes (they currently have them, but `loading="lazy"` could be added or `next/image` used for better optimization).

## ЁЯЯв MINOR (good-to-have)

- **`app/layout.tsx` (Line 2-20):**
  - **Issue:** Uses `next/font/google` correctly, but could explicitly set `preload: true` for at least the most critical font weight(s).
  - **Fix:** Confirm Next.js defaults to preloading, but explicit declaration helps clarity.

- **Framer Motion `m` Component:**
  - **Issue:** The app imports the full `motion` object from `framer-motion` everywhere.
  - **Fix:** Switch from `motion.div` to `m.div` and implement `<LazyMotion features={domAnimation}>` at the root/layout level to shave off ~25kb-30kb from the initial JS bundle.

- **`next.config.ts`:**
  - **Issue:** Empty configuration. No Bundle Analyzer configured.
  - **Fix:** Add `@next/bundle-analyzer` to measure layout weights and enable image domain configs if any external images are ever used.
